# Gin Auth — Complete Developer Guide

---

## What are we building?

A REST API that lets users **register**, **login**, and **view their profile**. Passwords are
hashed with bcrypt. Identity is proven with a JWT token sent on every protected request.

The stack:
- **Gin** — HTTP router and middleware
- **sqlc** — generates type-safe Go from raw SQL
- **pgx** — PostgreSQL driver
- **golang-jwt** — JWT creation and validation
- **bcrypt** — password hashing
- **golang-migrate** — database migrations

## How a request flows through the app

Understanding this flow is the most important thing. Every HTTP call travels through
exactly these layers in order:

```
HTTP Request
     │
     ▼
  Gin Router (api/routes.go)
     │  matches the path e.g. POST /api/v1/auth/login
     │
     ▼
  Middleware (internal/middleware/jwt.go)
     │  only runs on protected routes
     │  reads Authorization header, validates JWT, injects userID into context
     │
     ▼
  Handler (internal/handler/auth_handler.go)
     │  parses and validates the JSON request body
     │  calls the service
     │  writes the JSON response
     │
     ▼
  Service (internal/service/auth_service.go)
     │  contains all business logic
     │  hashes passwords, compares passwords, generates JWT tokens
     │  calls the repository for data
     │
     ▼
  Repository (internal/repository/user_repository.go)
     │  the only layer that talks to the database
     │  uses sqlc-generated functions
     │
     ▼
  Database (PostgreSQL)
```

Each layer only knows about the layer directly below it. The handler never touches
the database. The repository never knows about HTTP. This separation means you can
change your database without touching your handlers, or change your HTTP framework
without touching your business logic.

## Layer by layer — what each file does and why

---

### 1. `go.mod` — Module declaration

**What it does:** Declares your module name and lists all external packages your app needs.

**Key concept:** The module name (`gin-auth`) is used as a prefix in all your imports:
```go
import "gin-auth/internal/service"  // refers to internal/service/ in your project
```

**When you touch it:** When adding a new external dependency. After editing, always run:
```bash
go mod tidy
```

---

### 2. `.env.example` — Environment variables

**What it does:** Documents every environment variable the app needs. You copy this to `.env`
and fill in real values.

**Key concept:** The app never reads `.env` directly — Go reads from actual shell environment
variables. You must export them first:
```bash
export $(cat .env | xargs)
```

**Why not hardcode config?** So the same binary can run in development, staging, and production
with different databases and secrets — without changing any code.

---

### 3. `sqlc.yaml` — sqlc configuration

**What it does:** Tells sqlc three things:
1. Where your migration files live (your schema)
2. Where your SQL query files live
3. Where to write the generated Go code

**Key concept:** sqlc reads your `.sql` files and generates Go functions. You never write
the database code by hand — you write SQL, sqlc writes Go.

---

### 4. `db/migrations/` — Database schema

**What it does:** Plain SQL files that create (up) and destroy (down) your database tables.

**Key concept:** Migration files are numbered and never edited after they're applied.
If you need to change the schema, you create a new migration file:
```
001_create_users.up.sql     ← create users table
002_add_avatar_to_users.up.sql  ← add a column later
```

**`001_create_users.up.sql`** creates the `users` table:
```sql
CREATE TABLE IF NOT EXISTS users (
    id         BIGSERIAL    PRIMARY KEY,
    username   VARCHAR(32)  NOT NULL,
    email      VARCHAR(255) NOT NULL UNIQUE,  -- UNIQUE enforces no duplicate emails
    password   TEXT         NOT NULL,
    created_at TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);
```

Why `BIGSERIAL`? It auto-increments — you never set the ID yourself, PostgreSQL assigns it.
Why `TIMESTAMPTZ`? Stores timezone-aware timestamps — important for global apps.

---

### 5. `db/queries/users.sql` — SQL queries

**What it does:** Every SQL query your app will run, annotated with a name and return type
that sqlc uses to generate Go code.

```sql
-- name: CreateUser :one   ← ":one" means return a single row
INSERT INTO users (username, email, password, created_at, updated_at)
VALUES ($1, $2, $3, NOW(), NOW())
RETURNING *;               ← RETURNING gives us the inserted row (with the new ID)

-- name: GetUserByEmail :one
SELECT * FROM users WHERE email = $1 LIMIT 1;

-- name: GetUserByID :one
SELECT * FROM users WHERE id = $1 LIMIT 1;
```

**Annotation types:**
- `:one` → returns one row, Go function returns `(User, error)`
- `:many` → returns multiple rows, Go function returns `([]User, error)`
- `:exec` → returns no rows (for DELETE, UPDATE), Go function returns `error`

**After editing this file, always re-run:**
```bash
sqlc generate
```

---

### 6. `db/sqlc/` — Generated code (never edit manually)

**What it does:** sqlc generates three files from your migrations and queries:

- `db.go` — the `Queries` struct and `DBTX` interface (the database connection wrapper)
- `models.go` — Go structs that mirror your database tables
- `users.sql.go` — one Go function per query in `users.sql`

**What the generated code looks like:**
```go
// This is what sqlc generates from your SQL query:
func (q *Queries) GetUserByEmail(ctx context.Context, email string) (User, error) {
    row := q.db.QueryRow(ctx, getUserByEmail, email)
    var u User
    err := row.Scan(&u.ID, &u.Username, &u.Email, &u.Password, &u.CreatedAt, &u.UpdatedAt)
    return u, err
}
```

You get a fully type-safe function — no `interface{}`, no runtime errors from wrong column
names. If your SQL is wrong, `sqlc generate` fails at generate time, not at runtime.

---

### 7. `config/config.go` — Configuration

**What it does:** Reads environment variables and packages them into a typed `Config` struct.
Also builds the PostgreSQL DSN (Data Source Name) connection string.

**Key concept:** Every other package imports `config.Config` — nobody calls `os.Getenv()`
directly. This means there's one place to change how config is loaded.

```go
// DSN builds the connection string pgx needs:
// "host=localhost port=5432 user=postgres password=... dbname=gin_auth sslmode=disable"
func (c *Config) DSN() string {
    return fmt.Sprintf("host=%s port=%s user=%s password=%s dbname=%s sslmode=%s", ...)
}
```

---

### 8. `pkg/response/response.go` — Response envelope

**What it does:** Every API response — success or error — has the same shape:

```json
{
  "success": true,
  "message": "login successful",
  "data": { ... }
}
```

**Why a shared package?** Without this, every handler writes its own JSON structure.
Clients can never predict the response format. With this, every response is consistent.

```go
// In any handler:
response.Success(c, http.StatusOK, "login successful", resp)
response.Error(c, http.StatusUnauthorized, "invalid email or password")
```

---

### 9. `internal/model/user.go` — Domain model

**What it does:** Defines the Go structs that represent your data across the whole app.

**Three kinds of structs:**
```go
// User — the domain object, returned from service to handler
type User struct {
    ID        int64     `json:"id"`
    Email     string    `json:"email"`
    Password  string    `json:"-"`  // json:"-" means NEVER appear in JSON responses
    ...
}

// RegisterRequest — what the client sends to POST /register
type RegisterRequest struct {
    Email    string `json:"email"    binding:"required,email"`  // validated by Gin
    Password string `json:"password" binding:"required,min=6"`
}

// AuthResponse — what we send back after login/register
type AuthResponse struct {
    Token string `json:"token"`
    User  *User  `json:"user"`
}
```

**Key concept:** `json:"-"` on Password means even if you accidentally pass a User with
a password to `response.Success()`, the password hash will never appear in the output.

**Why separate from `db/sqlc/models.go`?** The sqlc model is tied to the database schema.
The domain model is what your business logic works with. Keeping them separate means
a database column rename doesn't break your API response format.

---

### 10. `internal/database/postgres.go` — Database connection

**What it does:** Opens a `pgxpool.Pool` — a pool of reusable database connections.

**Key concept:** A pool means your app doesn't open a new connection for every request
(expensive). Instead it borrows a connection from the pool, uses it, and returns it.

```go
func Connect(ctx context.Context, dsn string) (*pgxpool.Pool, error) {
    pool, err := pgxpool.New(ctx, dsn)
    pool.Ping(ctx)  // verify the connection actually works at startup
    return pool, nil
}
```

---

### 11. `internal/repository/user_repository.go` — Data access layer

**What it does:** The only layer that talks to the database. Uses sqlc-generated functions.
Translates database errors into your own sentinel errors.

**Key concept — the interface:**
```go
type UserRepository interface {
    Create(ctx context.Context, user *model.User) error
    FindByEmail(ctx context.Context, email string) (*model.User, error)
    FindByID(ctx context.Context, id int64) (*model.User, error)
}
```

The service depends on this *interface*, not the concrete struct. This means:
- In tests, you can pass a fake in-memory repository instead of a real database
- You can swap PostgreSQL for another database by writing a new struct that satisfies this interface
- The service never needs to change

**Sentinel errors:** Instead of leaking `pgx.ErrNoRows` up to the service, the repository
translates it:
```go
if errors.Is(err, pgx.ErrNoRows) {
    return nil, ErrNotFound  // clean, predictable error the service can check
}
```

**The `toModel()` function** converts a `db.sqlc.User` (database type) to `model.User`
(domain type). This is the boundary between the database world and the business logic world.

---

### 12. `internal/service/auth_service.go` — Business logic

**What it does:** Contains all the rules of the application. No HTTP, no SQL — pure logic.

**Register flow:**
1. Hash the password with bcrypt (never store plain text)
2. Call `repo.Create()` to save the user
3. If email is taken, return `ErrEmailTaken`
4. Generate a JWT token
5. Return the token + user

**Login flow:**
1. Find user by email (`repo.FindByEmail()`)
2. If not found, return `ErrInvalidCredentials` (same error as wrong password — prevents
   user enumeration: attackers can't tell if an email exists)
3. Compare the submitted password against the stored bcrypt hash
4. Generate a JWT token
5. Return the token + user

**JWT token:**
```go
// The token payload (Claims) embeds the user ID
type Claims struct {
    UserID int64 `json:"user_id"`
    jwt.RegisteredClaims  // contains ExpiresAt, IssuedAt
}
```

When a protected request comes in, we validate the token and extract `UserID` — that's
how we know who is making the request without hitting the database.

---

### 13. `internal/middleware/jwt.go` — Auth guard

**What it does:** A Gin middleware function that runs before protected handlers. If the
token is missing or invalid, it stops the request immediately (`c.Abort()`). If valid,
it injects the user ID into the request context so the handler can use it.

```go
// After middleware runs, any handler can get the user ID:
userID := c.MustGet(middleware.UserIDKey).(int64)
```

**Key concept:** Middleware is reusable. You attach it to a route group once:
```go
protected := v1.Group("/auth")
protected.Use(middleware.JWTAuth(authSvc))  // every route in this group is now protected
{
    protected.GET("/profile", authHandler.GetProfile)
    // add more protected routes here
}
```

---

### 14. `internal/handler/auth_handler.go` — HTTP handlers

**What it does:** Each function handles one endpoint. The pattern is always:
1. Parse and validate the request body (`c.ShouldBindJSON`)
2. Call the service
3. Map service errors to HTTP status codes
4. Write the response

```go
func (h *AuthHandler) Login(c *gin.Context) {
    // Step 1 — parse
    var req model.LoginRequest
    if err := c.ShouldBindJSON(&req); err != nil {
        response.Error(c, http.StatusBadRequest, err.Error())
        return
    }

    // Step 2 — call service
    resp, err := h.authSvc.Login(c.Request.Context(), &req)

    // Step 3 — map errors to status codes
    if errors.Is(err, service.ErrInvalidCredentials) {
        response.Error(c, http.StatusUnauthorized, err.Error())
        return
    }

    // Step 4 — write response
    response.Success(c, http.StatusOK, "login successful", resp)
}
```

**Key concept — `c.Request.Context()`:** Every service call passes the request context.
This allows database queries to be cancelled if the client disconnects mid-request,
preventing wasted work.

---

### 15. `api/routes.go` — Route registration

**What it does:** Mounts all routes onto the Gin engine and defines which middleware
applies to which routes.

```go
v1 := r.Group("/api/v1")

// Public — no auth needed
auth := v1.Group("/auth")
auth.POST("/register", authHandler.Register)
auth.POST("/login", authHandler.Login)

// Protected — JWT required
protected := v1.Group("/auth")
protected.Use(middleware.JWTAuth(authSvc))
protected.GET("/profile", authHandler.GetProfile)
```

**Why group routes?** So you can apply middleware (auth, rate limiting, logging) to
a whole group at once instead of each route individually.

---

### 16. `cmd/server/main.go` — Entry point

**What it does:** Wires all dependencies together and starts the server. This is the
only file that knows about all the other packages.

```go
// The wiring order matters — each step depends on the previous:
pool    := database.Connect(ctx, cfg.DSN())    // 1. DB connection
queries := db.New(pool)                         // 2. sqlc wraps the pool
userRepo := repository.NewUserRepository(queries) // 3. repo wraps sqlc
authSvc  := service.NewAuthService(userRepo, ...) // 4. service wraps repo
authHandler := handler.NewAuthHandler(authSvc)    // 5. handler wraps service
api.RegisterRoutes(r, authHandler, authSvc)       // 6. routes wire handler + middleware
r.Run(addr)                                       // 7. start listening
```

  -path db/migrations \

psql -U postgres -d gin_auth -c "\d users"

## How to add a new feature

Say you want to add a `DELETE /api/v1/auth/account` endpoint to delete the logged-in user's account.
Follow the layers bottom-up:

**1. Add the SQL query** (`db/queries/users.sql`):
```sql
-- name: DeleteUser :exec
DELETE FROM users WHERE id = $1;
```

**2. Regenerate sqlc:**
```bash
sqlc generate
```
This adds a `DeleteUser(ctx, id)` function to `db/sqlc/users.sql.go`.

**3. Add to the repository interface and implementation** (`internal/repository/user_repository.go`):
```go
// Add to interface:
Delete(ctx context.Context, id int64) error

// Add implementation:
func (r *userRepo) Delete(ctx context.Context, id int64) error {
    return r.q.DeleteUser(ctx, id)
}
```

**4. Add to the service** (`internal/service/auth_service.go`):
```go
// Add to interface:
DeleteAccount(ctx context.Context, userID int64) error

// Add implementation:
func (s *authService) DeleteAccount(ctx context.Context, userID int64) error {
    return s.repo.Delete(ctx, userID)
}
```

**5. Add the handler** (`internal/handler/auth_handler.go`):
```go
func (h *AuthHandler) DeleteAccount(c *gin.Context) {
    userID := c.MustGet(middleware.UserIDKey).(int64)
    if err := h.authSvc.DeleteAccount(c.Request.Context(), userID); err != nil {
        response.Error(c, http.StatusInternalServerError, "could not delete account")
        return
    }
    response.Success(c, http.StatusOK, "account deleted", nil)
}
```

**6. Register the route** (`api/routes.go`):
```go
protected.DELETE("/account", authHandler.DeleteAccount)
```

That's the complete pattern for every new feature.

## Common errors and fixes

**`sqlc: command not found`**
```bash
export PATH=$PATH:$(go env GOPATH)/bin
```

**`migrate: command not found`**
```bash
brew install golang-migrate
```

**`Database connection failed: pinging postgres`**
- PostgreSQL is not running → `brew services start postgresql`
- Wrong password in `.env` → check `DB_PASSWORD`
- Database doesn't exist → `psql -U postgres -c "CREATE DATABASE gin_auth;"`

**`relation "users" does not exist`**
You forgot to run the migration:
```bash
migrate -path db/migrations -database "postgres://..." up
```

**`401 authorization header required`**
You're hitting a protected route without the `Authorization: Bearer <token>` header.

**`401 invalid or expired token`**
The token has expired (default TTL is 24h) — log in again to get a fresh token.
